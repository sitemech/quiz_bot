
# NestQuiz - Отслеживание Проекта

## Роль: Исполнитель

### Задний план и Мотивация

1. Исправление бага: при выводе нескольких квизов описание выводилось только для первого события. Нужно выводить описание для каждого квиза под соответствующей информацией.
2. Улучшение команды `/start`: при получении команды `/start` нужно выводить не одно мероприятие, а все ближайшие мероприятия на сегодня и завтра, с описаниями, так же как выводится ежесуточно при наступлении времени по расписанию.
3. Исправление бага парсинга расписания: бот ошибочно пишет, что сегодня игры нет, хотя по URL с расписанием на сайте игра есть. Вероятно, на сайте поменялась верстка, и текущий парсер не находит матч на “сегодня”.

### Высокоуровневая Разбивка Задач

1. Исправить функцию `send_daily_notification()` для вывода описания каждого квиза
2. Изменить команду `/start` для вывода всех ближайших квизов на сегодня и завтра с описаниями
3. Адаптировать парсер расписания под изменения верстки сайта, чтобы корректно находились игры на сегодня

- [x] Изменить команду `/start` для вывода всех ближайших квизов с описаниями

### Обратная Связь Исполнителя или Запросы на Помощь

**Задача 1 выполнена:** Исправлена логика в функции `send_daily_notification()`. Теперь описание каждого квиза выводится сразу после информации о месте проведения соответствующего события.

**Задача 2 выполнена:** 
- Создана новая функция `format_upcoming_quizzes_message()` (строки 157-177), которая формирует сообщение о всех ближайших квизах на сегодня и завтра с описаниями
- Функция `send_daily_notification()` теперь использует эту новую функцию для избежания дублирования кода
- Команда `/start` (функция `subscribe()`, строки 193-203) теперь выводит все ближайшие квизы на сегодня и завтра с описаниями, используя ту же логику, что и ежедневные уведомления

**Изменения:**
- Вынесена общая логика формирования сообщения в функцию `format_upcoming_quizzes_message()`
- Команда `/start` теперь использует `format_upcoming_quizzes_message()` вместо `get_next_quiz()`
- Изменен текст сообщения при подписке: "Ищу ближайшие квизы..." вместо "Ищу ближайший квиз..."

### Уроки

## План (Planner) — Исправление парсинга расписания

### Цель
Сделать так, чтобы бот **корректно находил и выдавал игру на сегодня**, если она действительно есть на сайте по указанному URL.

### Предположения
- Точка отказа: HTML-верстка страницы расписания изменилась (классы/структура/тексты), и текущие CSS/XPath-селекторы или правила извлечения дат/времени перестали совпадать.
- Текущая бизнес-логика “что считать игрой на сегодня” остаётся прежней, ломается именно извлечение данных.

### Ключевые риски
- Даты на сайте могут быть в другом формате/локали, либо перенесены в `data-*` атрибуты/JSON внутри страницы.
- Сайт мог начать грузить расписание динамически (JS), и тогда простой `requests + BeautifulSoup` перестанет видеть матчи без выполнения JS.

### High-level Task Breakdown (минимальными шагами)

1) Локализовать место, где определяется “есть ли игра сегодня”
- **Что сделаем**: найдём функцию(и), которые ходят по URL расписания, парсят HTML и формируют вывод “игры нет/есть”.
- **Success criteria**: есть конкретный участок кода, который возвращает пустой результат “на сегодня” из-за несоответствия верстке.

2) Снять “снимок” актуальной страницы и понять новую структуру данных
- **Что сделаем**: получим HTML по URL (и при необходимости — запросы, которые делает страница), посмотрим где лежат дата/время/соперник/место.
- **Success criteria**: определён устойчивый источник данных (новые селекторы или JSON endpoint).

3) Обновить парсер под новую верстку (с fallback-логикой)
- **Что сделаем**:
  - обновим селекторы и правила извлечения;
  - добавим более “гибкий” поиск (например, по текстовым якорям и/или `data-*` атрибутам), чтобы следующий мелкий редизайн не ломал всё сразу.
- **Success criteria**: парсер извлекает минимум: дату, время, команды/название игры, и корректно классифицирует “сегодня”.

4) Добавить быстрый самотест на “сегодня” (без внешних зависимостей)
- **Что сделаем**: вынесем парсинг в чистую функцию (если ещё не вынесено) и добавим тест/проверку на реальном HTML-фрагменте (fixture), чтобы ловить будущие изменения.
- **Success criteria**: тест падает на старой логике и проходит после правки.

5) Ручная проверка сценария
- **Что сделаем**: прогон команды/функции, которая отвечает пользователю, и проверка, что **на текущую дату** игра показывается.
- **Success criteria**: бот выводит игру на сегодня (и не говорит “игры нет”), при условии что на сайте она есть.

### Запросы к пользователю (если понадобятся в Executor)
- Подтверждение URL расписания (если он берётся из конфига/переменной окружения) и пример “как выглядит правильный вывод” для игры на сегодня.
